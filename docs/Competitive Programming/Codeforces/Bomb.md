# Bomb

- 难度：<font color="#a0a">1979</font>
- 初见于： 2024/8/17 练习

## 分析
首先可以很容易地得到贪心策略，并写出 $\mathcal{O}(k\log{}n)$ 的做法：在 $k$ 次操作中，每次都选取最大的元素加入答案中，那么最终答案一定是最大的。为了实现 “选取最大的元素” 这一操作，需要使用树形结构，如 `multiset` 或 `priority_queue`。

问题在于， $k$ 最大可至 $10^9$，直接给上述做法宣判了死刑。我们需要找到某种不需要模拟 $k$ 次的算法，但又能同样起到模拟 $k$ 次的效果——这就是有趣的地方。

为了解题，需要一些 obseravation。我们的策略是 “选取最大的元素”，这意味着每当我们选取了元素 $x$，之前选取的元素一定大于等于 $x$，而剩余的元素一定小于等于 $x$！

因为我们要进行 $k$ 次操作，假设选取的第 $k$ 个数是 $x$，意味着经过 $k$ 次操作之后，数组 a 中的元素均小于等于 $x$。不妨转换思路，我们可以计算出 “使数组 a 中的元素均小于等于 $x$” 所需的最小操作次数 $f(x)$，如果这个操作次数小于等于 $k$，那么它就是合法的。

注意到，最小的操作次数 $f(x)$ 具有单调性。如果 $x\to\infty{}$ ，那么 $f(x)=0$。随着 $x$ 的不断减小，$f(x)$ 将越来越大，可能会超过 $k$。我们需要寻找的就是使得 $f(x)$ 恰好不超过 $k$ 的 $x^*$，它就是我们第 $k$ 次操作所选取的数（为什么？请你自己思考）。

正因为 $f(x)$ 具有单调性，所以可以使用二分来找到 $x^*$。剩下还有 $k-f(x^*)$ 次操作，它们所选取的元素也一定等于 $x^*$，否则就和 $x^*$ 的定义发生了矛盾。

## 代码
```cpp linenums=1
void solve() {
    int n = 0, k = 0;
    std::cin >> n >> k;

    std::vector<int> a(n), b(n);
    for (int i = 0; i < n; i++) {
        std::cin >> a[i];
    }
    for (int i = 0; i < n; i++) {
        std::cin >> b[i];
    }

    auto F = [&](int x) {
        i64 res = 0;
        for (int i = 0; i < n; i++) {
            if (a[i] >= x) {
                res += (a[i] - x + b[i] - 1) / b[i];
            }
        }
        return res;
    };

    int L = 0, R = *std::ranges::max_element(a) + 1, M = 0;
    while (L < R) {
        M = (L + R) >> 1;
        if (F(M) <= k) {
            R = M;
        } else {
            L = M + 1;
        }
    }

    i64 ans = 1LL * (k - F(L)) * L;
    for (int i = 0; i < n; i++) {
        if (a[i] >= L) {
            int op = (a[i] - L + b[i] - 1) / b[i];
            ans += 1LL * op * a[i] - 1LL * op * (op - 1) / 2 * b[i];
        }
    }

    std::cout << ans << '\n';
}
```

## 感悟
久违的 <font color="#a0a">1900*</font> 题。到目前为止我还没有做出过难度比这个更高的题目，而且已经做出的大多也是因为看过题解……水平远远不够。

这道题理论上应该初见于几个星期前的 Div.3，但我做到这题的时候，时间已经所剩无几，并没有在这道题上进行足够的思考，所以也就不算 “见过”。

我花了两天来解决这道题。暴力的 $\mathcal{O}(k)$ 做法很容易想到，但对于本题来说显然不够。

不知道思考了几个小时。我知道，正确的做法肯定不会暴力地模拟 $k$ 次操作，但不模拟的话又该如何做呢？

因为暴力做法使用的是优先队列，所以我试图想象操作优先队列（堆）的情景——取出堆顶元素，将元素值累加进答案中，再将修改后的元素放回堆中……放回后的元素可能依然位于堆顶，也可能不是。这个过程实在是太过于 “具体” 了！正因为如此，才难以对其抽象化，进而加快计算的速度。

实在不得已，我打开了 editorial。题解中写到，“使用二分搜索值 $x$，使得所有加入答案的值都大于等于 x”。我知道这是很关键的提示，但我完全摸不着头脑，为什么要这么突兀地去定义这样的 $x$？

又想了一个小时，无果，遂去百度其他人写的题解。

终于在某个 CSDN 博客里面找到了解释：如果我们当前加入答案的元素是 $x$，那么之前已经加入答案的元素的值肯定大于等于 $x$。直到这里我才理解官方题解的用意，并做出本题（当然，再没有看过题解）。

如果某人第一次遇到这种题目就能做出这样的 observation，于我而言他就是天才。

天才是很少的，因此我不认为那些 3 分钟秒杀的大佬们都是初见本题。他们一定是做过了很多很多类似的题目。

我知道了，当出现“取最大元素” 的操作时，并不总是需要模拟具体的操作，而是可以去枚举最后取的那个元素 $x$。在该操作的约束下，所有已取的元素都大于等于 $x$，所有未取的元素都小于等于 $x$。

下一次遇到类似的题目，绝不会再像这样陷入窘境。
